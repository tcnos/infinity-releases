Seven fixes and three new capabilities across managed lists, pipeline stages, fund team management, and discussions.

## Features

### [INVOS-40] User profile picture
Users can now upload a profile picture from Profile & Settings. The image is downscaled client-side and shown in the top-bar user menu (with an initials fallback). No object storage required — stored inline on the user record.

### [INVOS-62] Invite users to a fund by email
The fund **Users → Invite by email** action now works. It creates a pending user account if the email is new, then assigns them to the fund with the chosen role (idempotent for existing users). The endpoint had been lost during the Postgres rewrite while the button remained, so invites were silently failing.

## Enhancements

### [INVOS-64] Fund Users list sorted by name
The Pending and Active sections of a fund's Users list are now ordered alphabetically by name (case-insensitive), making larger teams easier to scan.

## Fixes

### [INVOS-33] Newly added users now appear in the Portfolio Manager dropdown
A user freshly added to a fund did not show up in the Deal Team → Portfolio Manager dropdown until a manual page refresh, because the candidate list was cached for five minutes. The dropdown now refreshes immediately whenever fund membership changes.

### [INVOS-41] Discussion pagination no longer flashes the new-post composer
Paging through discussions briefly showed the "new discussion" composer on every page change, because the list rendered empty while the next page loaded. The current page now stays visible until the next one resolves.

### [INVOS-45] New/removed company types appear in dropdowns without a refresh
Adding or removing a Company Type (or any managed list value) did not update the corresponding dropdown on a company's profile until the page was refreshed, due to a cache-key mismatch. Managed-list changes now propagate to all dropdowns immediately.

### [INVOS-57] Duplicate pipeline stages are now rejected
Users could create multiple stages with the same name (even differing only in case). Stage titles are now validated as unique within a fund and bucket, case-insensitively, with a DB index as a backstop.

### [INVOS-59] Managed-list values are now case-insensitive-unique
Managed lists (e.g. Company Type) accepted near-duplicate values like "Domestic" and "domestic". Values are now unique within their category, case-insensitively, enforced both in the API and at the database level.

### [INVOS-63] Admin "Edit → Active" now clears a user's Pending status
Activating an invited user via Admin → Users → Edit set them active under the hood but left the Pending badge in place, leaving invited users with no clear activation path. Activating a user now clears the pending state so the badge and the login gate agree.

## Deployment notes

**Run the incremental migration after pulling the new images — no data re-seed required:**
```bash
docker compose exec inf-backend sh -c "cd /app && npx knex migrate:latest"
docker compose restart inf-backend
```

Migrations applied:
- `013` — user avatar column (INVOS-40)
- `014` — case-insensitive unique indexes for managed lists and stages (INVOS-59/57). De-duplicates any existing case-variant rows before creating the indexes.
